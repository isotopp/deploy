# Deploy Refactoring Notes

## Goal

Replace the current single-file `deploy` script with a packaged Python application that:

- runs as `uv run --managed-python deploy ...`
- can be installed as `uv tool install .`
- preserves at least the current supported workflows
- improves structure, safety, validation, and testability
- supports future generalization beyond the current Apache-only setup where practical
- supports source trees from local Git checkouts and generic repository URLs, not only GitHub SSH URLs
- supports `--json` output for machine-readable workflows

## Current Script Summary

The current script is a 957-line single-file Python 3.11 program that:

- parses CLI arguments with `argparse`
- stores project definitions as JSON files in `/etc/projects/<project>`
- generates Apache config files in `/etc/httpd/conf.sites.d/<hostname>.conf`
- manages Apache SSL domain aggregation in `/etc/httpd/conf.d/ssldomain.conf`
- creates/removes Unix users for some project types
- creates SSH deploy keys
- clones GitHub repos
- sets up Python virtualenvs and installs requirements
- creates/deletes systemd units for `discord_bot`
- creates/deletes rsyslog config and shared logrotate config
- starts/stops/restarts services
- tails logs

Supported project types today:

- `static_site`
- `redirect_site`
- `wsgi_site`
- `discord_bot`
- `proxy`

Supported operations today:

- `create`
- `delete`
- `logs`
- `restart`
- `show`
- `start`
- `stop`
- `update`

## Current Live Apache Reality

The current deploy script is not the whole truth for the running Apache setup.

### Generated by deploy

The following site configs are currently simple generated deploy output:

- `grafana.home.koehntopp.de`
- `immich.home.koehntopp.de`
- `navidrome.home.koehntopp.de`
- `plik.home.koehntopp.de`
- `webauthn.home.koehntopp.de`

These map cleanly to the current narrow model:

- simple HTTP reverse proxy by local port
- simple WSGI app

### Hand-written outside deploy

The following site configs are present in Apache but are not represented in `/etc/projects` and are not generated by the current script:

- `hass.home.koehntopp.de`
- `kris.home.koehntopp.de`
- `openwebui.home.koehntopp.de`
- `unifi.home.koehntopp.de`
- `vaultwarden.home.koehntopp.de`

This matters because `/etc/projects` currently contains only:

- `grafana`
- `immich`
- `navidrome`
- `plik`
- `webauthn`

So `/etc/projects` is not currently the full source of truth for the Apache estate.

### Concrete shortcomings visible in live config

The current model fails as soon as a site needs anything beyond a trivial local HTTP proxy or fixed WSGI layout.

Examples from the live system:

- `unifi` needs HTTPS upstream proxying to `https://127.0.0.1:8443/` plus `SSLProxyEngine` and SSL peer-check overrides.
- `vaultwarden` needs mixed HTTP and websocket behavior, specifically a custom websocket route for `/notifications/hub`.
- `openwebui` needs path-specific routing for `/static`, `/_app`, `/api`, `/ws/socket.io`, `/c/...`, and `/auth`, plus custom rewrite rules and forwarding headers.
- `hass` proxies to a non-local upstream `192.168.1.39:8123`, not `127.0.0.1`, and has custom websocket handling for `/api/websocket`.
- `kris` is not a plain static site; it exposes multiple aliases into storage paths such as `/audiobook`, `/book`, `/music`, `/comic`, and `/video`, and also differs in `AllowOverride`.

### TLS/domain handling is also incomplete

The script currently builds `ssldomain.conf` from:

- the machine FQDN
- a hardcoded `SSLDOMAIN_LIST`
- hostnames found in `/etc/projects`

That is not sufficient as a long-term model:

- some live sites are neither in `/etc/projects` nor obviously modeled by the current deploy types
- the hardcoded list is already a maintenance smell
- certificate domain management is coupled to incomplete deployment metadata

This is a major reason the refactor should treat Apache site definition as a first-class typed model, not a side effect of a tiny project-type enum.

## What The Current Script Gets Right

- The operational model is simple and understandable.
- `/etc/projects` is a good source-of-truth boundary.
- Generated Apache configs are intentionally minimal and macro-based.
- The script already separates project types conceptually.
- A dry-run mode exists.

## Problems In The Current Design

### Structural issues

- All concerns live in one file: CLI, config parsing, validation, rendering, filesystem writes, process execution, service management, and logging setup.
- Most functions depend on global `args`.
- The code uses side effects heavily and returns little structured state.
- There is no library boundary; everything is command-oriented.

### Data model issues

- Project config is untyped JSON with ad-hoc validation.
- Type-specific fields are not modeled explicitly.
- Placeholder values such as `/fake_home` and `/fake_pubkey` leak into persisted config for non-user-backed project types.
- `/etc/projects` is not rich enough to describe several live Apache sites currently in production.
- source location handling is hardcoded around the legacy `github` field and `git@github.com:...` assumptions
- There are field inconsistencies and bugs:
  - `to_hostame` is misspelled when writing redirect project files.
  - validation checks `args.to_host`, but the CLI argument is `to_hostname`.
  - hostname validation comment is outdated and does not match actual allowed domains.

### Execution and safety issues

- Commands are run with `shell=True`.
- Some operations interpolate user-controlled values directly into shell commands.
- `git reset --hard` is always used during update.
- Restarts and file writes are not transactional.
- There is no explicit privilege boundary model even though many operations require root.
- Errors are mostly printed instead of raised and aggregated cleanly.

### Packaging/runtime issues

- Hardcoded Python path: `/usr/bin/python3.11`
- Hardcoded venv workflow instead of a modern package/runtime story
- No package metadata, entry points, dependency management, linting, or formatter config
- The runtime model is tied to the host Python instead of a managed `uv` Python

### Testing issues

- No tests
- No isolated rendering tests
- No filesystem abstraction
- No command runner abstraction to mock

## Refactoring Direction

The refactor should split the current script into a small CLI plus a testable application library.

### Proposed package shape

```text
deploy/
  pyproject.toml
  src/deploy/
    __init__.py
    cli.py
    settings.py
    models.py
    errors.py
    paths.py
    renderer.py
    runner.py
    project_store.py
    apache.py
    systemd.py
    users.py
    gitops.py
    logging_setup.py
    operations/
      __init__.py
      create.py
      delete.py
      update.py
      restart.py
      start.py
      stop.py
      show.py
      logs.py
  tests/
    ...
```

### Recommended tooling

- Packaging/runtime: `uv`
- CLI: `typer` or `click`
- Data modeling: `dataclasses`
- Validation: explicit typed validation code on top of dataclasses
- Templates: simple dedicated render functions first; `jinja2` only if rendering becomes materially more complex
- Testing: `pytest`
- Typing/checking: `ty`
- Formatting/linting: `ruff`
- HTTP client if needed: `httpx`

`typer` is a pragmatic fit here: it supports a much cleaner subcommand tree than raw `argparse`, which directly addresses the current CLI mess.

## Proposed Core Model

Use typed dataclass models instead of ad-hoc dictionaries.

### Base project model

Common fields:

- `name`
- `type`
- `hostname`
- `username | None`
- `home | None`
- `project_dir | None`
- `source | None`

Type-specific models:

- `StaticSiteProject`
- `RedirectProject`
- `ProxyProject`
- `WsgiAppProject`

Notes:

- Drop `discord_bot` from the refactored scope. It is conceptually separate from web-server/site management.
- Replace fake persisted values with optional fields.
- Store only semantically meaningful data in `/etc/projects`.
- Keep model construction separate from validation so compatibility loading from older JSON is still manageable.
- Treat legacy `github` as a compatibility input that maps to a more general `source` field.

### Revised site taxonomy

The live Apache config suggests the site model should be richer than the current four deployment shortcuts.

A more realistic phase-1 taxonomy is:

- `static_site`
- `redirect_site`
- `wsgi_site`
- `proxy_site`
- `custom_site`

Where:

- `proxy_site` covers reverse-proxied applications
- `custom_site` exists as an explicit escape hatch for Apache-managed sites that cannot yet be expressed through the higher-level model

This is preferable to pretending every real deployment can be reduced to `proxy(port)` or `wsgi(appdir)`.

### Proxy site submodel

`proxy_site` should not be modeled as only `hostname + port`.

It needs room for at least:

- upstream scheme: `http` or `https`
- upstream host
- upstream port
- preserve-host toggle
- request header policy
- encoded slash handling
- timeout knobs
- websocket routes
- path-specific proxy routes
- rewrite rules where needed

Without this, the tool will immediately fall back to hand-written Apache again.

## CLI Shape

The current CLI is one flat parser with a shared option bag. That should be replaced.

### Target command structure

Preferred shape:

- `deploy create <type> <name> [type-specific options]`
- `deploy delete <name>`
- `deploy show <name>`
- `deploy show projects`
- `deploy update <name>`
- `deploy restart <name>`
- `deploy start <name>`
- `deploy stop <name>`
- `deploy logs <name>`

Examples:

- `deploy create proxy immich --hostname immich.home.koehntopp.de --port 2283`
- `deploy create wsgi-site webauthn --hostname webauthn.home.koehntopp.de --repo git@github.com:isotopp/webauthn-test.git --username webauthn`
- `deploy create wsgi-site webauthn --hostname webauthn.home.koehntopp.de --source git@github.com:isotopp/webauthn-test.git --username webauthn`
- `deploy create redirect-site oldsite --hostname old.example.com --to-hostname new.example.com`

### CLI design requirements

- `create` must be a command group with type-specific subcommands or equivalent typed argument handling.
- Type-specific options should only appear on the relevant create path.
- Shared options such as `--dry-run`, `--verbose`, and path overrides should live at the top level.
- The help output should make the supported deployment shapes obvious without requiring the user to infer them from docs.

## Proposed Operation Model

The current script mixes project type with lifecycle behavior. The refactor should make that explicit.

### Separate concerns

- `ProjectStore`: read/write/list project definitions
- `Renderer`: turn project models into Apache/systemd/rsyslog text
- `Executor`: run commands with structured results
- `Provisioners`: create users, keys, checkouts, environments
- `Operations`: create/delete/update/start/stop/restart/show/logs

### Plugin-style handlers

Use per-project-type handlers:

- `ProxySiteHandler`
- `RedirectHandler`
- `StaticSiteHandler`
- `WsgiHandler`
- `CustomSiteHandler`

Each handler should declare:

- required fields
- generated artifacts
- service behavior
- update behavior
- log behavior

That gives a controlled path to add future types without piling more `if/elif` into one file.

## Apache Generalization

The immediate target is still Apache management, but the design should avoid baking Apache-specific assumptions into the whole application.

### Keep Apache as a backend module

- `apache.py` should own Apache-specific path defaults and rendering
- project handlers should request generated artifacts, not write files directly
- the CLI should still expose a single `deploy` command

### Preserve current config shape initially

Keep generating the same minimal macro-based Apache snippets:

- `Use VHost ...`
- `Use ProxyVHost ...`
- `Use RedirectVHost ...`
- `Use PyApp ...`

This minimizes rollout risk.

But the live system shows that this is not enough as the end state. The refactor should preserve current simple output first, while leaving room for:

- proxying to non-local upstreams
- HTTPS upstreams
- per-path proxy rules
- per-site alias mappings
- custom websocket routes
- richer header and rewrite behavior

### Generalize where useful

Potential generalizations:

- allow multiple virtual hosts per project later
- allow per-project TLS/domain options instead of global hardcoded lists
- allow alternate web backends in the future
- allow app runners other than `main.py` + venv assumptions
- allow Apache-managed custom sites to be represented explicitly instead of drifting outside the deploy system

Do not over-generalize in phase 1. Preserve behavior first.

## Packaging Plan

### `pyproject.toml`

Add:

- project metadata
- dependencies
- optional dev dependencies
- console script entry point, for example:

```toml
[project.scripts]
deploy = "deploy.cli:app"
```

### Installation modes

Target both:

- `uv run --managed-python deploy ...`
- `uv tool install .`

### Runtime compatibility

- Use `uv` managed Python rather than the system interpreter
- Pin an explicit minimum supported Python in `pyproject.toml`
- Assume the active interpreter may be newer than the server's system Python
- Remove hardcoded interpreter path usage
- Prefer `sys.executable` where an interpreter path is needed
- If a managed interpreter version needs to be tracked in-repo, add the corresponding `uv` metadata rather than relying on `/usr/bin/python3.11`

## Filesystem and Command Execution

Introduce explicit abstractions.

### Filesystem

Encapsulate:

- atomic file writes
- chmod/chown
- symlink creation
- directory creation
- path existence checks

### Command runner

Encapsulate:

- subprocess execution without `shell=True` by default
- timeouts
- environment
- user/group switching when needed
- structured return values

This is critical for both security and testability.

## Logging and Output

Current output is print-driven. Refactor to:

- use structured exceptions internally
- present user-friendly CLI output at the edge
- keep `--dry-run`
- add `--verbose`
- support machine-readable output via `--json`
- ensure help text is hierarchical and type-specific rather than one flat list of mostly irrelevant flags

## Migration Strategy

### Phase 1: behavior-preserving extraction

- create package skeleton
- move constants and path defaults into settings
- add typed dataclass project models
- implement project store against current `/etc/projects` JSON format
- extract Apache rendering functions that reproduce current output exactly
- extract systemd/rsyslog rendering functions
- add command runner abstraction
- implement the new hierarchical CLI surface
- implement `show` and rendering tests first

### Phase 2: operation parity

- implement `create`, `delete`, `update`, `start`, `stop`, `restart`, `logs`
- preserve current CLI surface where possible
- support compatibility aliases:
  - `discord_bot` -> `service`
- preserve current project file format compatibility on read

### Phase 3: safer semantics

- remove fake persisted values
- fix redirect field naming
- make update strategy configurable
- add atomic config writes
- validate before writing
- support config diff/dry-run output

### Phase 4: controlled generalization

- separate web/app/service deployment concepts more clearly
- add richer service definitions
- add multiple runtime strategies if needed
- add richer Apache options if needed

## Testing Strategy

Tests should be built from the start, not bolted on later.

### Unit tests

Test:

- project model validation
- compatibility loading of existing `/etc/projects/*` JSON
- compatibility loading of legacy `github` source definitions
- Apache rendering for all project types
- systemd rendering
- rsyslog/logrotate rendering
- SSL domain aggregation logic
- GitHub URL conversion
- CLI parsing for the new `create <type> <name>` command tree

### Filesystem tests

Use `tmp_path` to test:

- writing project definitions
- writing generated configs
- creating symlinks
- dry-run behavior

### Command runner tests

Mock subprocess execution and assert:

- correct argv
- correct timeouts
- correct user/group handling
- error propagation

### Operation tests

Use fake filesystem + mocked runner to test:

- `create` for each supported project type
- `delete` cleanup behavior
- `update` behavior with and without repo/venv
- `restart` behavior and SSL domain regeneration

### Golden-file tests

Use golden snapshots for generated Apache/systemd config so output drift is explicit.

### Optional integration tests

Later, add integration tests that run in a disposable temp tree with overridable paths:

- fake `/etc/projects`
- fake `/etc/httpd/conf.sites.d`
- fake `/etc/systemd/system`

This requires path injection via settings, which should be designed in early.

### Tooling commands

Planned developer workflow:

- tests: `pytest`
- type checking: `ty`
- format `src` and `tests`: `ruff format`
- lint `src` with autofix: `ruff check --fix src`

## Immediate Compatibility Requirements

These behaviors should remain supported in the first refactor release:

- read current JSON files from `/etc/projects`
- generate the same Apache snippets for current known projects
- keep `show`, `restart`, and `update` behavior available
- keep `proxy`, `redirect_site`, `static_site`, and `wsgi_site`
- expose creation through the new `create <type> <name>` CLI shape, even if read compatibility still accepts legacy type names internally
- accept both legacy `github` metadata and newer generalized `source` metadata

And the design should explicitly acknowledge that some current live sites cannot yet be losslessly imported from the existing `/etc/projects` model.

## Specific Code Issues To Fix During Refactor

- remove global `args`
- fix redirect option naming mismatch
- fix persisted `to_hostame` typo
- stop persisting fake fields for proxy/redirect projects
- stop using `shell=True` for normal execution
- stop hardcoding `/usr/bin/python3.11`
- stop assuming Python apps always run as `main.py`
- make `httpd` restart behavior testable and idempotent
- make project file loading stricter and error messages clearer
- remove the flat create parser in favor of typed subcommands
- stop pretending `/etc/projects` currently represents every Apache site on the machine

## Suggested First Deliverable

The first implementation milestone should be:

- package skeleton
- `pyproject.toml`
- typed dataclass models
- settings/path injection
- project store
- Apache renderer
- hierarchical CLI with `create <type> <name>`
- `show` command
- tests for parsing and rendering

That is enough to establish the architecture before touching destructive lifecycle operations.

After that, the next milestone should be a design for representing the currently hand-written Apache sites, especially:

- HTTPS upstream proxying
- non-local upstreams
- path-based proxy routing
- websocket-specific routing
- alias-heavy static/document sites
- explicit custom Apache sites where abstraction would otherwise become forced and brittle

## Recommendation

Refactor in-place into a real package rather than trying to gradually beautify the current single-file script.

The old script already contains enough mixed concerns and compatibility hacks that incremental cleanup inside the current shape will keep the design muddy. A new package with compatibility-focused readers and renderers is the cleaner path.
